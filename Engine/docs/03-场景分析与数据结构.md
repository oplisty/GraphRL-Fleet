## 4. 数据结构设计（答辩重点）

核心选型：

1. 邻接表图：`Graph.adj: dict[int, list[Edge]]`
2. 边哈希索引：`edge_lookup[(u,v)] -> distance`（O(1) 查边）
3. 对象哈希索引：`vehicles/tasks/stations: dict[id, obj]`
4. 任务状态集合：`pending/assigned/completed/expired_task_ids: set`
5. 充电队列：`ChargingStation.queue: deque`
6. 充电并发槽：`charging_slots: list[int|None]`
7. 车辆路线：`route + route_index + distance_to_next`
8. 最短路缓存：`distance_cache/path_cache + source shortest-path tree cache`
9. 协同分配映射：`Task.assigned_vehicles: dict[vehicle_id, load]`
10. 已配送累计：`Task.delivered_weight / delivered_by_vehicle`

为什么这样选：

- 路网稀疏，邻接表比邻接矩阵更省空间。
- 高频查边需要哈希索引，不走线性遍历。
- 任务状态集合避免每个 step 扫全量任务。
- `deque` 能直接表达 FIFO 排队与负荷压力。
- 路径缓存降低调度评估时的重复最短路成本。

复杂度（核心）：

- `get_vehicle_state`：`O(1)`
- `get_available_tasks`：`O(P)` (`P` 为 pending 数)
- `dispatch`（含最短路）：`O((V+E)logV)`，缓存命中后可显著降低
- 选站 `_choose_reachable_station`：`O(S * shortest_path_query)`

课程可讲亮点：

1. 邻接表 + 哈希边索引组合设计
2. 状态集合化管理
3. 充电队列 + 多桩槽位建模
4. 路径缓存与源点最短路树缓存
5. 状态机与结构化日志保证可解释性

## 5. 协同任务如何开启

默认不开启（保持旧行为）：

- `collaborative_task_ratio: 0.0`

示例（开启 30% 协同任务）：

```bash
python -m Framework.examples.run_baseline \
  --config "Framework/configs/random_baseline.yaml" \
  --collaborative-task-ratio 0.3
```
